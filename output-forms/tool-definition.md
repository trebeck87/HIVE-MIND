# Output Form: Tool Definition

The shape of a tool/function schema generated by the colony.

---

## Structure

```json
{
  "name": "[tool_name — snake_case, descriptive]",
  "description": "[1-2 sentences: what it does + when to use it + when NOT to use it]",
  "input_schema": {
    "type": "object",
    "properties": {
      "[param_name]": {
        "type": "[string|number|boolean|array|object]",
        "description": "[What this param does — include valid values, ranges, defaults]"
      }
    },
    "required": ["[list required params]"]
  }
}
```

## Design Principles

**Description must include WHEN and WHEN NOT:**
```
❌ "Searches the web for information"
✅ "Searches the web for current information not in training data. Use for: real-time data, recent events, fact-checking. Do NOT use for: general knowledge questions, historical facts, coding help."
```

**Parameters need constraints, not just types:**
```
❌ "query": { "type": "string", "description": "Search query" }
✅ "query": { "type": "string", "description": "Search query. 1-6 words for best results. Start broad, narrow based on results. Do not use quotes or operators." }
```

**Include examples in description when behavior is non-obvious:**
```
"description": "Date filter in ISO format. Example: '2025-01-15T00:00:00Z'. Use 'today' for current date."
```

## Companion Prompt Instructions

Every tool definition should be accompanied by usage instructions in the system prompt:

```
<tool_selection_criteria>
- Use [tool_name] when: [specific conditions]
- Do NOT use [tool_name] when: [specific conditions]
- If [tool_name] fails: [fallback behavior]
- Maximum [N] calls to [tool_name] before reporting to user
</tool_selection_criteria>
```
